# 差押予定一覧 管理ツール（seizure_list_manager）GUI版 機能要件書

## 1. ツールの目的

deposit_seizure_list / insurance_seizure_list が生成した CSV（預金差押予定一覧.csv / 生命保険差押予定一覧.csv）を閲覧・検索し、登録内容の編集および差押中止となった案件の「引抜」操作（処理済フラグの設定による WinActor 処理対象からの除外）を安全に行うためのツール。

現在は特定の1名が CSV を手作業で編集しており、行ずれ・列ずれ・誤上書きのリスクと属人化が課題となっている。本ツールにより、全職員が GUI 上で安全に編集・引抜操作を実行できるようにする。

## 3. 起動方式

- seizure_list_manager.exe をダブルクリックで起動する。
- 起動時に seizure_list_manager_config.json を読み込み、参照先の deposit / insurance の config から CSV パスを解決する。
- 起動時に両 CSV を一括メモリ読み込みする。
- デフォルトで「預金」タブがアクティブ。
- 引数は不要。

- ConfirmOverlay と同一レベルの Grid 重ね合わせ。同一の半透明背景（#CCFFFFFF）。

- 通常時は Visibility.Collapsed。

## 5. タブ切替

### 5.1 タブ構成
- 2タブ：「預金」「生命保険」。UniformGrid Rows='1' HorizontalAlignment='Left' で両タブを均等幅にする。
- タブ Padding='12,8,12,6'。
- アクティブタブ：#546E7A テキスト + Medium + Border の下ボーダー2px（`BorderThickness='0,0,0,2'` + `BorderBrush='#546E7A'`。file_search のサイドパネルタブと同一方式）。
- 非アクティブタブ：#777 テキスト + Normal + `BorderBrush='Transparent'`。
- タブフォントサイズ：11px。

### 5.2 タブ切替時の動作
- テーブルの表示データを切り替える（預金 CSV ⇔ 生保 CSV）。
- 検索ボックスのテキストをクリアし、フィルタを解除する。
- 列指定ドロップダウンの選択を「宛名番号」（デフォルト）にリセットする。
- 「引抜済みを非表示」トグルの状態は維持する。
- ソート状態はリセットし、デフォルト（文書番号降順）に戻す。
- フッターの CSV パス表示を切り替える。
- フッターの件数情報を切り替えたタブのデータで更新する。

### 5.3 CSV 未検出時
- 参照先の config が存在しない、または outputFolder 配下に CSV ファイルが存在しない場合、該当タブは無効化（グレーアウト）する。
- 両方とも未検出の場合、テーブルエリアに「CSV ファイルが見つかりません」メッセージを表示する。

## 6. 検索フィルタ

### 6.1 構成
- 検索ボックス（左側、Width='400' 固定）と列指定ドロップダウン（右側、コンパクト）を結合表示する。
- 検索ボックス: file_search の検索ボックスと同一の XAML パターンを使用する（Border + Grid 重ね合わせ、Ellipse+Line の虫眼鏡アイコン、Padding 28,8、BorderBrush #D0D0D0、CornerRadius 4）。
- プレースホルダ：「検索...」（#999、テキスト入力時に非表示）。

### 6.2 列指定ドロップダウン
- 検索ボックスの右端に結合（border-radius: 左0、右6px）。
- 選択肢：「宛名番号」（デフォルト）、「氏名」、「処分担当」、「文書番号」。
- 11px / コンパクト幅（選択肢テキストが収まる最小幅）。
- 変更時にフィルタを再適用する。

### 6.3 即時フィルタ動作
- テキスト入力のたびに、選択中の列を対象に部分一致フィルタを適用する。
- マッチしない行は非表示（Visibility.Collapsed）となる。
- フッターの件数情報がリアルタイムで更新される。
- 「引抜済みを非表示」トグルとの組み合わせは AND 条件で動作する。
- フィルタはメモリ上のデータに対して行うため、ファイル I/O は発生しない。

### 6.4 × クリアボタン
- テキストがあるときのみ表示。
- クリック時：テキストクリア → フィルタ解除（全件表示）→ 検索ボックスにフォーカス戻し。
- ControlTemplate 方式（file_search と同一パターン）。
- 丸背景（20px, CornerRadius=10）に × アイコン（Line 2本、StrokeStartLineCap/EndLineCap='Round'）。
- 通常: 背景 #E0E0E0 / アイコン #777。ホバー: 背景 #CCC / アイコン #555。
- Focusable=False。

- ウィンドウリサイズ時・タブ切替時に SizeChanged イベントで列幅を再計算する。

- 支店名列: 預金タブ 120px / 生保タブ 0px（非表示）。
- 氏名列が残幅を埋める（最小 80px）。

### 7.4 ListViewItem スタイル
- Grid 重ね合わせ方式：rowBd（背景 Border）+ accent（3px 左端アクセントバー）。
- 通常: 白背景、下線 #F0F0F0。
- ホバー: #F0F3F5 背景。
- 選択: #E8EEF0 背景 + #546E7A アクセントバー。
- 引抜済み行: 7.6 を参照。
- GridViewRowPresenter Margin='0,7,0,7'（行パディング、他ツールと統一）。

### 7.5 ソート
- 列ヘッダークリックでソート切替（昇順 ▲ / 降順 ▼）。
- デフォルト：文書番号の降順（最新の文書番号が先頭）。
- ソートインジケータ（▲/▼）をヘッダーテキストに付与。
- ソート状態はタブごとに独立して保持する。
- 引抜済み行もソート対象に含む（非表示トグル OFF 時）。

### 7.6 引抜済み行の表示
- 処理済フラグ1 の値が「2」の行を引抜済みとして扱う。
- 「引抜」列に「済」テキストを表示（#BF360C / 12px / Medium）。
- 行全体の背景色を #FAFAFA に変更。
- 行全体のテキスト色を #B0B0B0 に変更。
- 引抜済み行は選択可能だが、「引抜」ボタンは無効（既に引抜済みのため）。

### 7.7 複数選択
- Ctrl+クリック / Shift+クリックによる複数行選択をサポートする。
- 複数選択時、引抜済みでない行が1件以上含まれていれば「引抜」ボタンは有効。
- 引抜実行時、選択行のうち引抜済みでない行のみを処理対象とする。

### 7.8 コンテキストメニュー
- 右クリックで表示。
- 「編集」：選択行の編集オーバーレイを表示（9. 編集操作と同じ動作）。引抜済みでない行が1件だけ選択されている場合のみ有効。
- 「引抜」：選択行を引抜（8. 引抜操作と同じ動作）。引抜済み行のみ選択時は無効。

### 7.9 空白エリアクリック
- テーブルの空白エリアをクリックした場合、選択を解除する。
- VisualTree を辿って ListViewItem が見つからなければ空白エリアと判定する。

### 7.10 ダブルクリック
- データ行のダブルクリックで編集オーバーレイを表示する（9. 編集操作参照）。
- VisualTree を辿り、ListViewItem 上のクリックであることを確認する（列ヘッダーのダブルクリックでは発火しない）。
- 引抜済みでない行が1件だけ選択されている場合のみ有効。

### 7.11 印刷済み行の表示
- 処理済フラグ1 の値が「1」の行を印刷済みとして扱う（WinActor 決裁用調書シナリオが印刷完了時に設定する。正本シナリオはフラグ2 を使用し、フラグ1 との間にタイムラグがある）。
- 「印刷」列に「済」テキストを表示（#107C41 / 12px / Medium）。
- 行のグレーアウトは行わない（グレーアウトは引抜済み専用。印刷済みは正常に工程が進んだ状態のため）。
- 引抜・編集の操作は可能。ただし引抜は警告付き（8.2 参照）、編集は注意書き付き（9.4 参照）。
- 引抜実行後はフラグ1 が「2」に上書きされるため、「印刷」列の「済」は消え、「引抜」列の「済」表示に置き換わる（印刷済みだった履歴は CSV 上に残らない。引抜後の紙の後始末は担当職員への口頭連絡で運用する）。
- 預金・生保の両タブで同一の実装（生保で印刷工程が未運用の間は列が空のままとなる）。
- 「印刷済みを非表示」フィルタで表示対象から除外できる（8.6 参照）。

## 8. 引抜操作

### 8.1 引抜ボタン
- アクションバー右端に配置。
- プライマリスタイル（#546E7A 塗り、白テキスト）。
- ボタンテキスト: `'↓ 引抜'`（単一 TextBlock、Padding='0,0,4,0' で左オフセット）。deposit の `'＋ 一覧に追加'` と同一パターン。
- 有効条件：テーブルで引抜済みでない行が1件以上選択されていること。
- 無効時：背景 #CCC、テキスト #999。

### 8.2 引抜確認オーバーレイ
- 引抜ボタンクリック時に ConfirmOverlay をフェードインで表示する。
- 半透明白背景（`Background='#CCFFFFFF'`）。deposit_seizure_list のオーバーレイと同一色。
- オーバーレイの範囲: ヘッダーバーとフッターバーを除く全領域（タブバー・ツールバー・テーブル・アクションバーを覆う）。deposit_seizure_list と同じ DockPanel 構造。
- 中央配置のカード（白背景 / CornerRadius 8 / Padding 24,20 / DropShadowEffect BlurRadius=16 ShadowDepth=4 Opacity=0.12）。
- カードの MaxWidth: 「宛名番号0123456789（執行日: 令和00年00月00日）の登録を引き抜きます」が1行で収まる幅に設定。
- フェードインアニメーション: Opacity 0→1、150ms。
- フェードイン後、キャンセルボタンにフォーカスを移す（Enter 連打での再発火防止）。
- カードの構成:
  - 上部：確認メッセージ
    - 単数選択時：「宛名番号 XXXXXXXXXX（執行日: 令和X年Y月Z日）の登録を引き抜きます」
    - 複数選択時：「N 件の登録を引き抜きます」
  - 中部：補足テキスト（11px / #777）：「処理対象から除外されます」（句点なし）
  - 中部（印刷済み警告・条件付き表示）：選択対象に印刷済み（フラグ1=1）の行が含まれる場合のみ表示（11px / #D32F2F / TextWrapping）
    - 単数選択時：「この行は印刷済みです。引抜後、担当職員へ連絡してください。」
    - 複数選択時：「M 件は印刷済みです。引抜後、担当職員へ連絡してください。」（M = 対象中の印刷済み件数）
  - 下部：ボタン2つ（右寄せ）
    - 「キャンセル」ゴーストボタン
    - 「実行」プライマリボタン

### 8.3 CSV 書き込み処理
- 確認オーバーレイで「実行」をクリックした場合に実行する。
- 対象行の処理済フラグ1 と処理済フラグ2 に「2」を設定する。
- 処理済フラグ3 は変更しない。
- 行の削除は行わない（文書番号の発番が狂うため）。
- CSV ファイルの排他ロック内で「読み込み→フラグ書き換え→書き戻し」をアトミックに実行する。
- リトライ機構（最大5回、500ms間隔）を備える。
- 書き込み後、CSV をディスクから再読み込みする（他ユーザーの同時変更も反映）。検索テキスト・ソート順は維持。
- CSV 書き込みの詳細は「13. CSV 書き込み」を参照。

### 8.4 書き込み後の視覚的更新
- 引抜処理完了後、確認オーバーレイをフェードアウト（Opacity 1→0、150ms）で閉じる。
- 結果オーバーレイは表示しない（テーブル行の視覚変化で即時フィードバックする）。
- 対象行を引抜済み表示に即座に更新する（7.6 の表示ルール適用。CSV 再読み込みにより他ユーザーの変更も反映）。
- フッターの件数情報を更新する。
- 「引抜済みを非表示」トグルが ON の場合、引き抜いた行は即座に非表示となる。
- 書き込み失敗時：確認オーバーレイをフェードアウトで閉じた後、エラーメッセージを表示する。メモリ上のデータは変更しない。

### 8.5 引抜済みフィルタ
- アクションバー左側にトグルスイッチを配置。
- ラベル：「引抜済みを非表示」（11px / #777）。
- OFF（デフォルト）：全件表示。
- ON：引抜済み行を表示対象から除外する（FilterRecords でスキップ）。
- 検索フィルタ・印刷済みフィルタとの組み合わせは AND 条件。
- トグル切替時にフッターの件数情報を更新する。
- トグルの状態はタブ切替時にも維持する。

### 8.6 印刷済みフィルタ
- 「引抜済みを非表示」トグルの右に配置。
- ラベル：「印刷済みを非表示」（11px / #777）。
- OFF（デフォルト）：全件表示。
- ON：印刷済み（フラグ1=1）の行を表示対象から除外する（FilterRecords でスキップ）。ON にすると未印刷の行のみが残るため、WinActor の処理待ち案件の確認に使える。
- 検索フィルタ・引抜済みフィルタとの組み合わせは AND 条件。
- トグル切替時にフッターの件数情報を更新する。
- トグルの状態はタブ切替時にも維持する。
- 留意点: 本フィルタ ON の状態で印刷済み行を引き抜くと、フラグ1 が 1→2 に変わり印刷済みではなくなるため、引抜済み行として再表示される（引抜済みフィルタも ON の場合は非表示のまま）。

## 13. CSV 書き込み

### 13.1 書き込み方式
- 追記方式ではなく全体書き換え方式を採用する（行の中間位置のフィールドを変更するため）。
- 排他ロック（FileShare.None）で CSV ファイルを開き、ロックを保持したまま以下をアトミックに実行する:
  1. 全行を読み込む
  2. 読み込んだ元の内容を `.bak` ファイルに書き出す（本体破損時の復旧元。詳細は 13.6）
  3. メモリ上で対象行のフィールドを書き換える
  4. ファイルポインタを先頭に戻し、全行を書き戻す
  5. ファイルを閉じ（ロック解放）、`.bak` を削除する
- 他の職員が同時に操作を行った場合、後発の操作はロック取得待ち（リトライ）となり、先発の変更が反映済みのファイルを読み直してから処理するため、変更の上書き消失は発生しない。
- 書き戻し（ステップ4）の途中でクラッシュ・ネットワーク断が発生しても、`.bak` に元の内容が保持されているため復旧できる。
- 引抜操作（WriteWithdrawalFlags）と編集操作（WriteEditedFields）は共通コア（WriteCsvWithBackup）を共用し、対象行の書き換え処理のみをそれぞれが定義する。

### 13.2 書き換え対象フィールド

#### 引抜操作
- 処理済フラグ1：「2」を設定。
- 処理済フラグ2：「2」を設定。
- 処理済フラグ3：変更しない。
- 上記以外の全フィールドも変更しない（元の値を保持）。

#### 編集操作
- CSV列[2]（氏名）：入力値で上書き。
- CSV列[3]（職員名）：入力値で上書き。
- CSV列[4]（執行日）：yyyy/MM/dd → 7桁和暦に変換して上書き。
- CSV列[5]（住民票住所）：入力値で上書き。
- CSV列[6]（届出住所）：チェック ON →「（届出：...）」でラップして上書き / チェック OFF → 空文字で上書き。
- 上記以外の全フィールドも変更しない（元の値を保持）。

### 13.3 排他ロック + リトライ
- FileStream を FileShare.None で開く（13.1 のステップ1〜4の全体がロック範囲）。
- リトライは失敗した段階に応じて2種類に分かれる:
  - **書き戻し開始前の失敗**（ロック競合・`.bak` 書き出し失敗）: ファイルを開き直し、読み込みからやり直す（最大5回、500ms間隔）。
  - **書き戻し中の失敗**: ファイルを読み直さず、メモリ上の行データで書き込みのみを再試行する（最大5回、500ms間隔）。切り詰められた本体を読み直して `.bak` を上書きする事故を防ぐため。
- 全リトライ失敗時はエラーメッセージ（バックアップからの復旧案内を含む）を表示し、メモリ上のデータは変更しない。書き戻し中の失敗の場合、`.bak` はそのまま残す。
- 書き戻し開始後は、いかなる失敗経路（ファイルクローズ時の Flush 失敗を含む）でも読み込みからのやり直しを行わない（writeStarted フラグで制御）。切り詰められた本体を読み直して `.bak` を上書きする事故を防ぐため。

### 13.4 エンコーディング
- BOM付きUTF-8（deposit_seizure_list / insurance_seizure_list の出力と一致）。

### 13.5 CSV 整合性の保持
- ヘッダー行を保持する。
- 行の順序を変更しない。
- フィールドのエスケープ規則は RFC 4180 に準拠する（読み込み時のエスケープ状態を維持）。
- 行の追加・削除は行わない。

### 13.6 バックアップと復旧

#### .bak ファイルの役割
- パス: 本体 CSV と同じフォルダの `{CSVファイル名}.bak`。
- 書き戻し前の元の内容を保持する（BOM付きUTF-8、本体と同一形式）。
- 書き込み成功時に削除されるため、**`.bak` の残存 = 前回の書き込みが未完了**であることを示すマーカーとして機能する。

#### 破損診断（CheckAndOfferRecovery）
- CSV 読み込み（LoadCsvRecords）の冒頭で `.bak` の存在を確認する。存在しない場合は何もしない（通常時のコストはゼロ）。
- `.bak` が存在する場合、本体 CSV の破損を機械的に診断する。書き戻しは先頭から順次進むため、破損は必ず末尾の欠落の形をとることを利用し、以下のいずれかに該当すれば破損と判定する:
  1. 本体ファイルが存在しない、または読み込めない
  2. ヘッダー行が欠落している、または `.bak` のヘッダーと不一致
  3. 最終データ行の列数が必要列数（預金20列 / 生保21列）に満たない
  4. 本体のデータ行数が `.bak` のデータ行数より少ない（書き込みは行数不変・他ツールは追記のみのため、正常時は減らない）
- 本体のロック中（他プロセスが書き込み中）は診断を見送り、次回の読み込み時に再確認する。

#### 診断結果ごとの動作
- **本体が健全**: `.bak` を黙って削除する（書き戻し開始前のクラッシュ等。復旧すると最新の編集が失われるため復旧は提案しない）。
- **本体が破損 + `.bak` が健全**: 復旧ダイアログ（MessageBox / はい・いいえ・キャンセル）を表示する。判断材料として、対象ファイル名・両ファイルの行数・最終更新日時を提示する。
  - はい: `.bak` から本体へコピー（File.Copy）して復旧し、`.bak` を削除する。コピー方式のため、復旧中にクラッシュしても `.bak` が無傷で残り再試行できる。
  - いいえ: 本体をそのまま使い、`.bak` を削除する。
  - キャンセル: 両ファイルを保持する（次回の読み込み時に再度確認する。判断に迷った場合の相談用の逃げ道）。
- **本体が破損 + `.bak` も不健全**: 自動復旧は提案せず、警告メッセージのみ表示する。両ファイルを保持する。

#### 既知の制約（残余リスク）
- 本体破損後、復旧が行われる前に deposit_seizure_list / insurance_seizure_list が破損した CSV に追記した場合、`.bak` から復旧すると追記された登録が失われる。破損行自体は読み込み時の列数不足警告（skipCount）で顕在化する。deposit / insurance 側への `.bak` 存在チェックの追加は将来の改修候補とする。
- 同一行を複数の職員が同時に「編集」した場合、後発の保存が先発の5フィールドを上書きする（last-writer-wins）。引抜操作は両者とも同じ値を書き込むため影響しない。

## 14. 和暦変換

### 14.1 era_mapping.json
- ツール同梱の era_mapping.json を使用する（deposit_seizure_list / insurance_seizure_list と同一フォーマット）。
- 起動時に exe と同一フォルダから読み込む。

### 14.2 変換ロジック
- CSV の執行日（7桁和暦: 元号コード + 2桁年 + 2桁月 + 2桁日）を人が読める形式に変換する。
- 表示形式：「令和8年7月15日」（ゼロ埋めなし）。
- 変換に失敗した場合は元の7桁和暦をそのまま表示する。
